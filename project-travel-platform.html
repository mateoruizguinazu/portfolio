One Codebase, Every Brand

A case study on architecting a white-label travel-booking platform — a single frontend that ships a fully-branded, fully-localized product for every client, with brand, language, and features driven entirely by configuration rather than code.

Project Overview

I led the ground-up rewrite of a travel-booking frontend into a true white-label platform: a single React monorepo codebase that deploys a distinct, fully-branded experience for multiple enterprise insurance and travel clients. Each client's identity — its theme, language, feature set, and analytics — is injected at build time, so standing up a new brand is a matter of configuration, not new code. Working across the boundary between design and engineering, I built both the design-token system that powers per-brand theming and the architectural conventions that keep the product consistent as it scales across clients.

The Challenge

The product needed to serve several enterprise clients at once, each with its own brand identity, language — including full right-to-left support — and feature requirements. The instinctive approach, forking or hard-coding the app per client, doesn't scale: every new brand multiplies the maintenance surface with duplicated UI, drifting design decisions, and brittle one-off theming. The real challenge was to design a single system where brand, locale, and feature availability are all configuration rather than code, while enforcing an architecture strict enough that quality wouldn't erode as the team and the client list grew.

The Solution

I architected the platform around three ideas that turn "another client" from a rewrite into a checklist.

Configuration-driven white-labeling

Every client is expressed as configuration — theme tokens, feature flags, routes, locale, and analytics. Switching the build mode produces a completely branded build of the same application, and onboarding a new client follows a typed checklist the compiler enforces, with no changes to product code.

A two-layer design-token system

Raw brand primitives — colour scales, typeface, corner radii — feed a layer of automatically-derived semantic tokens. Components only ever reference intent ("primary action," "surface," "body text"), never raw values, so an entirely new brand restyles the whole application by changing its primitives alone. Dark mode and right-to-left layouts are built into the same system rather than bolted on per client.

An architecture that defends itself

Library boundaries are enforced by lint rules, features and routes register through typed contracts, and an AI-assisted, spec-driven workflow — documented conventions, design briefs, and automated code review — keeps every contribution consistent. The result is a codebase that stays reviewable and coherent as it grows.
